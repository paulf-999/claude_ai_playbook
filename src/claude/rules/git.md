# 🌿 Rules — Git

## 📝 Commits

- Do not create commits unless explicitly asked.
- Format: `type(scope): imperative description` (Conventional Commits)
  - `type`: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`, `ci`, etc.
  - `scope`: the affected area (e.g., `dbt`, `pre-commit`, `makefile`)
  - Description: lowercase, imperative, no trailing period
  - Examples: `feat(pre-commit): add ruff`, `fix(dbt): correct model reference`
- When explicitly instructed, `--no-verify` is acceptable (e.g., to bypass pre-commit hooks during debugging). Do not use it by default.
- Stage specific files by name — never `git add -A` or `git add .` without review.
- Group related changes into a single commit where it makes sense — one commit per logical change, not one commit per file.
- Always pass commit messages via heredoc to preserve formatting.

## 🌿 Branch standards

### 🔒 Protected branches

- `main` is the long-lived production branch. Never commit to it directly.
- All work goes through short-lived `feature/`, `hotfix/`, or `release/` branches, merged back into `main` via PR.

### 🏷️ Branch naming

- Prefix: `feature/` for new features, `hotfix/` for urgent production fixes, `release/` for bulk changes spanning 20+ files (see PR limits below)
- Name: lowercase only, letters/numbers/underscores — no hyphens, spaces, or special characters
- Pattern: `^(feature|hotfix|release)/[a-z0-9_]+$`
- Examples: `feature/add_pr_template`, `hotfix/fix_commitlint`, `release/new_col_all_int_dim_merchant_models`

## 🗂️ Repo structure

New repos should follow the standard team template at [payroc/dmt-scripts-git_repo_template](https://github.com/payroc/dmt-scripts-git_repo_template). The expected top-level layout is:

```
.github/
  CODEOWNERS                  # code ownership assignments
  ISSUE_TEMPLATE/             # GitHub issue templates
  pull_request_template.md    # standard PR template
docs/                         # supporting documentation
src/                          # all source code and scripts
  make/                       # Makefile includes
  sh/                         # shell scripts and utilities
  templates/                  # reusable script/config templates
.gitignore
.pre-commit-config.yaml
.yamllint
CONTRIBUTING.md
Makefile
README.md
requirements.txt              # Python dependencies (if applicable)
```

- Do not place source code or scripts at the repo root — all code belongs under `src/`.
- Do not place documentation at the repo root (beyond `README.md` and `CONTRIBUTING.md`) — all docs belong under `docs/`.
- When creating a new repo, start from the template rather than from scratch.

## 🚫 .gitignore

- Every repo must have a `.gitignore` file.
- Base the `.gitignore` on the industry-standard templates for the languages and tools in use — [github.com/github/gitignore](https://github.com/github/gitignore) is the canonical source. For example, use the Python template for Python repos, Terraform template for Terraform repos, and compose multiple templates when a repo spans more than one language or tool.
- Review and update the `.gitignore` whenever the contents of the repo change — new languages, tools, editors, or build artefacts introduced to the repo should be reflected in `.gitignore` immediately.
- At minimum, always ignore:
  - Virtual environments (`venv/`, `.venv/`)
  - Environment files (`.env`, `.env.*`)
  - Build and cache artefacts (`__pycache__/`, `*.pyc`, `.terraform/`, `dist/`, `*.egg-info/`)
  - IDE and editor directories (`.idea/`, `.vscode/`)
  - OS-generated files (`.DS_Store`, `Thumbs.db`)
- Do not commit generated or ephemeral files — if something would be regenerated by a build or install step, it should be ignored.

## 🔍 Code review

- Every PR must be approved by at least one other team member before it can be merged.
- If the change affects a file or folder listed in `CODEOWNERS`, approval from the assigned code owner is mandatory — GitHub auto-assigns them as reviewers.
- Reviews should focus on clarity, correctness, and consistency with team conventions.

## 📋 Pull requests

- Keep PRs small and focused — split into multiple PRs if needed.
- PRs must contain fewer than 20 files. If a change genuinely cannot be split, use a `release/` branch prefix and ensure the PR applies the same type of change consistently (e.g., a bulk formatting or linting pass). The PR must be tied to a significant release and must be easy to review despite its size.
- PRs should follow the standard PR template with:
  - Summary of the change
  - Type: Feature / Trivial / Bugfix / Refactoring / Documentation / Breaking Change
  - Related Jira issue(s)
  - Testing checklist (DEV/local, UAT, other)
- Flag breaking changes explicitly and describe the rollout impact.
- Files must not contain merge conflict markers.
