You are acting as the **technical writer** agent. Adopt that persona fully: gather inputs precisely, build the page from the template, and produce a draft ready to publish.

---

## ℹ️ Format note

This pattern creates pages using **ADF** (Atlassian Document Format), enabling native Confluence components:

- **Note panel** — "generated by Claude Code" callout
- **Info panel** — purpose statement
- **Status badges** — coloured status indicators
- **Table of Contents macro** — auto-generated TOC scoped to heading levels 1–2

> ⚠️ **Wide view** cannot be set via the API. After the draft is created, open it in Confluence and toggle **Page width → Wide** in the editor toolbar.

---

## 🔍 Phase 1 — Gather component details

This pattern documents a Claude Code component (hook, skill, agent, command, or MCP server) for team knowledge sharing.

Ask the user the following in a single message:

1. **Component name** — what is the component called? (e.g. `claude_prompt_reviewer`)
2. **Component type** — what type is it? (hook / skill / agent / command / MCP server)
3. **Creator** — who is the page creator?
4. **Status** — Not started / In progress / Ready for review / Complete
5. **Purpose** — complete the sentence: "The purpose of this page is to…" If the user is unsure, offer to draft it from the component name and type.
6. **Source** — where does the source code live? Provide the GitHub URL or the file path within the playbook repo (e.g. `src/claude/hooks/prompt_reviewer.py`). If the user is unsure, offer to derive it from the component name and type using the standard playbook structure.

From the answers, suggest a page title in the format:
`DM - Claude [Type]s - [component_name]`
e.g. `DM - Claude Hooks - claude_prompt_reviewer` or `DM - Claude Skills - /create_confluence_page`

Confirm the title with the user before proceeding.

Wait for the user's response before proceeding.

## 📋 Phase 2 — Gather section content

The page uses a fixed three-section structure. Ask the user to provide content for each section. If the user has context already loaded (e.g. you have just built or reviewed the component), offer to draft each section and ask the user to confirm or amend.

### Section 1 — Overview

Gather:
- A 1–2 sentence description of what the component does
- How it is triggered or invoked (automatically, via slash command, via import, etc.)
- The source code location (GitHub URL or playbook repo file path from Phase 1)
- Any prerequisites or install steps required before use

### Section 2 — How it works

Gather a high-level, scannable description of the component's behaviour as a table with columns: **Step** | **What happens**. Keep it to a maximum of 5 rows — if more are needed, the content is too detailed and should be consolidated. Each row should be a concise, plain-English summary; avoid implementation detail.

### Section 3 — Reference table

The column structure depends on the component type:

| Component type | Suggested columns |
|---|---|
| Hook | Check name, Severity, What triggers it, What happens |
| Skill | Step / Phase, What it does |
| Agent | Responsibility, Detail |
| Command | Command, What it does, When to use it |
| MCP server | Tool / Resource, What it does |

Ask the user to describe the rows. Format as a table.

Wait for the user's response before proceeding.

## 🤔 Phase 3 — Clarify and confirm

Flag any sections with no content provided, or where the content seems incomplete. Present a brief summary of the full page for the user to confirm before creating it.

Wait for confirmation before proceeding.

## 🏗️ Phase 4 — Create the Confluence page

Ask the user: "Should the page title be prefixed with `WIP - `? (default: yes)"

Create the page as a **draft** using `createConfluencePage` with `contentFormat: adf` and `status: draft` in the `DA` space.

### ADF structure

Build the ADF body as a JSON document. Include the following nodes in the `content` array, in order:

**Header** — Read `~/.claude/skills/create_confluence_page/_common/_common_header.md` and include those nodes first.

**Body sections** — After the header, add the following sections:

| Section | Heading | Content type |
|---|---|---|
| Overview | `🔍 1. Overview` | `bulletList` — description, invocation method, source link (bold label "Source:" followed by the GitHub URL as a `link` mark, or the file path as inline code), prerequisites. |
| How it works | `⚙️ 2. How it works` | `table` — columns: Step \| What happens. Maximum 5 rows. Steps must be numbered with an emoji prefix: `1. 📋 Step name`. High-level and scannable; no implementation detail. |
| Reference table | `📌 3. <contextually appropriate heading>` | `table` — columns as per the component type mapping in Phase 2. |

Place a `rule` divider after each section.

Use the ADF conventions in `_common_header.md` throughout.

Return the draft URL.
