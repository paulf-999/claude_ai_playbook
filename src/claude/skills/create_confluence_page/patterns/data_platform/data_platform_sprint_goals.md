You are acting as the **technical writer** agent. Adopt that persona fully: gather inputs precisely, build the page from the template, and produce a draft ready to publish.

---

## ℹ️ Format note

This pattern creates pages using **ADF** (Atlassian Document Format), enabling native Confluence components:

- **Note panel** — "generated by Claude Code" callout
- **Info panel** — purpose statement
- **Status badges** — coloured status indicators
- **Table of Contents macro** — auto-generated TOC scoped to heading levels 1–2

> ⚠️ **Wide view** cannot be set via the API. After the draft is created, open it in Confluence and toggle **Page width → Wide** in the editor toolbar.

## 🔍 Phase 1 — Gather sprint details

Ask the user the following in a single message:

1. What is the sprint number for the new page?
2. Who is the page creator?

The page status is always **In progress** for a new sprint goals page. The purpose statement is auto-drafted as: "The purpose of this page is to outline the Data Management engineering commitments for Sprint [number], highlighting capacity, risk, and cross-team dependencies."

Confirm both with the user before proceeding.

---

## 📋 Phase 2 — Establish sprint content

Ask the user the following in a single message:

1. **Must items** — for each committed item, collect:
   - Initiative name
   - Theme (e.g. `Platform Stability & BAU`, `Engineering Alignment`, `Platform Strategy`)
   - Status (planned / in progress / waiting for X)
   - Key tasks (bullet points)
   - Jira ticket(s) — leave blank if none
   - Owner (DM team member)
   - Stakeholders
   - Why (1–2 sentences)
   - Sprint Outcome (single sentence)

2. **Should items** — same fields as Must.

3. **Blocked items** — same fields, but replace Sprint Outcome with Comments.

Wait for the user's response before proceeding.

---

## 🤔 Phase 3 — Clarify and confirm

Flag any items missing required fields (Owner, Why, Sprint Outcome / Comments). Present a summary table grouped by section (Must / Should / Blocked). Ask the user to confirm before creating the page.

Wait for confirmation before proceeding.

---

## 🏗️ Phase 4 — Create the Confluence page

Ask the user: "Should the page title be prefixed with `WIP - `? (default: yes)"

The page title format is: `DM Sprint [number] Goals` (e.g. `DM Sprint 42 Goals`). Apply the `WIP - ` prefix if confirmed.

Create the page as a **draft** using `createConfluencePage` with `contentFormat: adf` and `status: draft` in the `DA` space.

### ADF structure

Build the ADF body as a JSON document. Include the following nodes in the `content` array, in order:

**Header** — Read `~/.claude/skills/create_confluence_page/_common/_common_header.md` and include those nodes first.

**Body sections** — After the header, add the following sections:

| Section | Heading | Content type |
|---|---|---|
| Must | `✅ 1. Must (Committed this sprint)` | `table` — columns: Initiative \| Theme \| Status \| Key Tasks \| Jira \| Owner \| Stakeholders \| Why \| Sprint Outcome. One row per Must item. |
| Should | `📋 2. Should (Important, not sprint-critical)` | `table` — same columns as Must. One row per Should item. |
| Blocked | `🚧 3. Blocked` | `table` — columns: Theme \| Initiative \| Status \| Key Tasks \| Jira \| Owner \| Stakeholders \| Why \| Comments. One row per Blocked item. |

Place a `rule` divider after each section.

For Jira ticket references: render each ticket as a hyperlink using a `link` mark — `{"type": "link", "attrs": {"href": "https://payroc.atlassian.net/browse/<ticket-id>"}}`. If no ticket, leave the cell with an empty paragraph.

Key tasks within a table cell should use a `bulletList` node nested inside the `tableCell`.

Use the ADF conventions in `_common_header.md` throughout.

Return the draft URL. Note any items where Jira tickets are missing.
