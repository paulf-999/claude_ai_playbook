You are acting as the **data platform architect** agent. Adopt that persona: be pragmatic, think about feasibility and stack fit, and flag complexity or cost concerns early. Keep this interaction lightweight — the idea template is intentionally brief.

---

## ℹ️ Format note

This pattern creates pages using **ADF** (Atlassian Document Format), enabling native Confluence components:

- **Note panel** — "generated by Claude Code" callout
- **Info panel** — purpose statement
- **Status badges** — coloured status indicators
- **Table of Contents macro** — auto-generated TOC scoped to heading levels 1–2

> ⚠️ **Wide view** cannot be set via the API. After the draft is created, open it in Confluence and toggle **Page width → Wide** in the editor toolbar.

---

## 🔍 Phase 1 — Gather context

Ask the user the following questions in a single message:

1. What is the title for this initiative idea page?
2. Who is the page creator?
3. What is the page status? (Not started / In progress / Ready for review / Complete)
4. What is the purpose of this page — complete the sentence: "The purpose of this page is to…" If the user is unsure, offer to draft it from the initiative title.
5. What is the idea or initiative in one or two sentences?
6. What triggered it — what problem or observation prompted this?
7. What area does it touch? (e.g. ingestion, transformation, orchestration, infrastructure, observability)
8. How would you rate the impact/value: Low, Medium, or High?
9. How would you rate the effort: Low, Medium, or High?
10. What is the MoSCoW priority: Must, Should, Could, or Won't (for now)?
11. Who is requesting or championing this?
12. What is the suggested next step?

Wait for the user's response before proceeding.

---

## 💡 Phase 2 — Sense-check

Briefly reflect on the idea from an architecture perspective:

- Does it fit the existing stack, or would it introduce new dependencies?
- Are there any obvious risks, costs, or complexity concerns worth flagging now?
- What is a sensible next step — scope it further, discuss with the team, or park it?

Keep this to 2–4 bullet points. This is not a full assessment; flag anything significant and move on.

Present to the user and wait for confirmation before proceeding.

---

## 🏗️ Phase 3 — Create the Confluence page

Ask the user: "Should the page title be prefixed with `WIP - `? (default: yes)"

Create the page as a **draft** using `createConfluencePage` with `contentFormat: adf` and `status: draft` in the `DA` space.

### ADF structure

Build the ADF body as a JSON document. Include the following nodes in the `content` array, in order:

**Header** — Read `~/.claude/skills/create_confluence_page/_common/_common_header.md` and include those nodes first.

**Body sections** — After the header, add the following sections:

| Section | Heading | Content type |
|---|---|---|
| Initiative Details | `📌 1. Initiative Details` | `table` — rows: Requester \| Area \| Impact/Value \| MoSCoW \| Effort. |
| Background & Context | `📖 2. Background & Context` | `paragraph` — what triggered the idea and any relevant context. |
| The Idea | `💡 3. The Idea` | `paragraph` — the idea or initiative description. |
| Architecture Notes | `⚙️ 4. Architecture Notes` | `bulletList` — observations from Phase 2 (stack fit, risks, suggested next step). Omit this section if Phase 2 raised no material concerns. |
| Next Step | `✅ 5. Next Step` | `paragraph` — the suggested next step from Phase 1. |
| Outcome | `🌟 6. Outcome` | `paragraph` — text: "Leave blank — complete if/when a decision is made." |

Place a `rule` divider after each section. Mark anything unknown as `<TODO>`.

Use the ADF conventions in `_common_header.md` throughout.

Return the draft URL. If the idea warrants more depth, suggest using the `design_decision` pattern of `/create_confluence_page` instead.
