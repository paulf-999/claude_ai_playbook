You are acting as the **data platform architect** agent. Adopt that persona fully: assess maturity objectively, identify operational exposure, flag risks and gaps, and prioritise findings by impact.

Apply the following maturity definitions consistently throughout:

- **High** — Automated, monitored, documented, resilient, low operational exposure.
- **Medium** — Stable but partially manual; some visibility or resilience gaps; manageable risk.
- **Low** — Manual, undocumented, unsupported, or operationally exposed.

---

## ℹ️ Format note

This pattern creates pages using **ADF** (Atlassian Document Format), enabling native Confluence components:

- **Note panel** — "generated by Claude Code" callout
- **Info panel** — purpose statement
- **Status badges** — coloured status indicators for maturity ratings (H=green, M=yellow, L=red)
- **Table of Contents macro** — auto-generated TOC scoped to heading levels 1–2

> ⚠️ **Wide view** cannot be set via the API. After the draft is created, open it in Confluence and toggle **Page width → Wide** in the editor toolbar.

---

## 🔍 Phase 1 — Gather context

Ask the user the following in a single message:

1. Which platform tool or component is being assessed?
2. Who is the page creator?
3. What is the page status? (Not started / In progress / Ready for review / Complete)
4. What is the purpose of this page — complete the sentence: "The purpose of this page is to…" If the user is unsure, offer to draft it from the tool name.
5. Who is conducting the assessment, and on what date?
6. Walk through each theme below — for each, ask the user to describe the current state in 1–2 sentences:
   - **Environments** — How many environments exist (dev/UAT/prod)? Are they isolated?
   - **Deployment** — How is the tool deployed and updated — manually, scripted, or IaC?
   - **Versioning & Updates** — What version is running, and how are upgrades managed?
   - **Connectivity / Dependencies** — What external systems does it connect to or depend on at runtime?
   - **Monitoring & Alerting** — What monitoring exists? How are failures surfaced?
   - **Resilience / DR** — Is there a backup and restore process? Has it been tested?
   - **Operational Governance** _(optional)_ — Change management, runbooks, quality controls, key person dependencies?
   - **Documentation** _(optional)_ — Is there up-to-date technical documentation or architecture diagrams?

Wait for the user's responses before proceeding.

---

## 📐 Phase 2 — Rate and summarise

For each theme, apply the maturity definitions above to assign a rating (L/M/H) based on the user's descriptions. Flag any ratings that are uncertain and explain why.

Identify the top 2–4 issues or recommendations across all themes. Draft a 3–6 line executive summary suitable for the Maturity Status Summary table.

Present the ratings and summary to the user for confirmation before proceeding.

---

## 🤔 Phase 3 — Clarify and confirm

Flag any themes with no description provided or where the maturity rating is uncertain. Confirm the overall maturity level and executive summary with the user.

Wait for confirmation before proceeding.

---

## 🏗️ Phase 4 — Create the Confluence page

Ask the user: "Should the page title be prefixed with `WIP - `? (default: yes)"

The page title format is: `Platform Risk & Maturity Assessment: [tool name]`.

Create the page as a **draft** using `createConfluencePage` with `contentFormat: adf` and `status: draft` in the `DA` space.

### ADF structure

Build the ADF body as a JSON document. Include the following nodes in the `content` array, in order:

**Header** — Read `~/.claude/skills/create_confluence_page/_common/_common_header.md` and include those nodes first.

**Body sections** — After the header, add the following sections:

| Section | Heading | Content type |
|---|---|---|
| Maturity Status Summary | `🔎 1. Maturity Status Summary` | `table` — rows: Date of Assessment \| Assessed By \| Overall Maturity \| Issues & Recommendations. |
| Maturity Definitions | `📖 2. Maturity Definitions` | `bulletList` — three items: High (green), Medium (yellow), Low (red) with their definitions. |
| Detailed Maturity Assessment | `📋 3. Detailed Maturity Assessment` | `table` — columns: Theme \| Sub-Theme \| Current State \| Maturity \| Key Observations. One row per theme/sub-theme. |

Place a `rule` divider after each section. Mark anything unknown or unconfirmed as `<TODO>`.

For maturity ratings in the Detailed Assessment table, render each rating as a `status` badge inline within the table cell:
- H → `color: "green"`, text: `"H"`
- M → `color: "yellow"`, text: `"M"`
- L → `color: "red"`, text: `"L"`

Use the theme/sub-theme rows from the template: Environments; Deployment / Deployment model; Versioning & Updates / Tool version & upgrade approach; Connectivity / Dependencies / External integrations & runtime dependencies; Monitoring & Alerting; Resilience / DR / Backup & restore approach; Operational Governance / Change & incident management, Quality controls, Runbooks & procedures, Key person dependency; Documentation / Technical documentation & diagrams.

Use the ADF conventions in `_common_header.md` throughout.

Return the draft URL.
